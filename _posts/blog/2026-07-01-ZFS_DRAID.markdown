## 들어가기 앞서

  대규모 스토리지 환경에서 디스크 장애는 불가피한 사건입니다. 때문에 장애 발생 후 데이터 복구가 완료되기까지의 시간은 스토리지의 가용성과 안전성을 결정하는 핵심 지표입니다. 특히 디스크 수가 수십~수백 개를 넘어가는 대형 어레이에서는 장애 복구 시간이 수십 시간에 달하기도 하며, 이 기간 동안 추가 장애가 발생하면 데이터 손실 위험이 급격히 높아집니다.

  OpenZFS의 전통적인 RAID 구현인 `raidz2`는 검증된 안정성을 제공하지만, 대형 어레이에서 장애 복구 시 파일시스템 메타데이터 전체를 순회해야 하는 구조적 한계를 가지고 있습니다. 이를 극복하기 위해 OpenZFS 2.1.0에서 도입된 것이 `dRAID(Distributed RAID)`[^1]입니다.

  이번 글에서는 `dRAID`의 내부 구조를 간단한 소스코드 수준에서 분석하고, 90개 디스크로 약 `1PB(PetaByte)` 용량의 환경에서 직접 측정한 `raidz2-baseline`과 `dRAID2:9d:90c:1s`의 성능 비교 결과를 공유합니다.

## dRAID 구조

  `dRAID`를 이해하려면 먼저 그 내부를 구성하는 네 가지 개념, 행/열/그룹/슬라이스를 이해해야 합니다. 이 개념들은 OpenZFS 소스코드 `vdev_draid.c`[^2]에 직접 정의되어 있습니다.

### 행(Row)과 열(Column)

  `vdev_draid.c`는 행을 다음과 같이 정의합니다.

> `module/zfs/vdev_draid.c` Line : 65–67 <br>
> "First, let's define a 'row' in the configuration to be a 16M chunk from each physical drive at the same offset. This is the minimum allowable size since it must be possible to store a full 16M block when there is only a single data column."

  **행(Row)** 은 모든 물리 디스크의 동일한 오프셋에 위치한 `16MB` 블록들의 집합입니다. 행의 너비는 물리 디스크 수 `C`개로 고정되며, 데이터가 쌓일수록 행이 아래로 추가됩니다. `16MB`라는 크기는 ZFS의 최대 `recordsize`가 `16MB`이기 때문입니다. `data column`이 단 1개뿐인 극단적인 구성에서도 블록 하나를 행 하나에 온전히 담을 수 있어야 하므로, 이를 보장하기 위한 하한값입니다.

  **열(Column)** 은 물리 디스크 1개에 해당합니다. 예시로 `9d:90c:1s` 구성에서 `90c`가 열의 수이며, 어떤 상황에서도 변하지 않습니다.

### 그룹(Group)

  **그룹(Group)** 은 `D`개의 `data` 블록과 `P`개의 `parity` 블록으로 구성된 실제 `RAID` 보호 단위입니다. `dRAID2:9d:90c:1s` 기준으로 1그룹 = 9d + 2p = 11개 블록입니다. 그룹은 특정 디스크에 고정되지 않습니다. 슬라이스마다 순열이 바뀌어 다른 디스크 조합에 배치됩니다. 또한 소스코드에 명시된 것처럼 행 경계를 가로질러 배치될 수 있습니다.

> `module/zfs/vdev_draid.c` Line : 69–70 <br>
> "We allow groups to span multiple rows in order to align any group size to any number of physical drives."

  예시의 `dRAID2` 설정은 `90c:1s` 이기 때문에 90개 컬럼 중 `spare`를 제외한 89개를 11개씩 채우면 89 ÷ 11 = 8.09…로 행에 딱 맞지 않습니다. 따라서 `Row N`의 마지막이 어떤 그룹의 중간이라면, `Row N+1`의 첫 블록이 그 그룹의 이어지는 부분으로 간주됩니다.

### 슬라이스(Slice)

  **슬라이스(Slice)** 는 그룹들이 나머지 없이 딱 떨어지는 행의 묶음입니다. 슬라이스 경계마다 `permutation`이 바뀌어 다음 슬라이스에서는 `data/parity/spare`가 다른 물리 디스크 조합에 배치됩니다. 이 슬라이스당 행의 수 `R`은 최소공배수(LCM)로 계산합니다.

```
// module/zfs/vdev_draid.c Line : 71–78
R = LCM(D+P, C-S) / (D+P)
```

  이를 가지고 `9d:90c:1s (dRAID2)` 기준으로 정리하면 아래와 같습니다.

| 항목             | 값                   |
| :------------- | :------------------ |
| D+P            | 9+2 = 11            |
| C-S            | 90-1 = 89           |
| LCM(11, 89)    | 979 (11과 89는 서로소)   |
| R (슬라이스당 행 수)  | 979 / 11 = **89행**  |
| 슬라이스 당 전체 블록 수 | 89 * 90 = **8010개** |

## Spare의 분산 구조

  `dRAID`의 핵심 특징인 `distributed spare`는 일반적으로 "모든 디스크에 골고루 분산된다"고 알려져 있습니다. 그러나 소스코드를 분석하면 그 의미가 보다 구체적입니다. 해당 내용을 보면 `spare`의 위치는 다음과 같이 결정됩니다.

  먼저 관련 주석에는 이렇게 명시되어 있습니다.

> `module/zfs/vdev_draid.c` Line : 2337–2338 <br>
> "The last N columns of the dRAID permutation table are used to determine on which dRAID children a specific offset should be written."

즉 `spare`는 `permutation table`의 **마지막 S개 컬럼**으로 항상 고정 배치됩니다. 나머지 C-S=89개 컬럼이 그룹(data+parity)으로 순차적으로 채워지는 구조입니다. 그리고 이 메커니즘이 실제로 동작하는 부분은 `vdev_t * vdev_draid_spare_get_child` 의 아래 코드입니다.

```c
// module/zfs/vdev_draid.c L2418–2419
uint64_t cid = vdev_draid_permute_id(vdc, base, iter,
    (tvd->vdev_children - 1) - vds->vds_spare_id);
```

  `permutation` 결과에서 마지막 인덱스 `(children - 1) - spare_id`가 `spare`를 담당할 물리 디스크를 결정합니다.

<p align="center">
  <img src="/assets/draid/fig2_spare_per_slice.png" alt="dRAID 스페어 분산 구조" width="800"/>
  <br/>
  <em>dRAID 스페어 분산 구조</em>
</p>

  여기서 중요한 점이 있습니다. **슬라이스 안의 모든 행에서 spare를 담당하는 물리 디스크는 동일합니다.** 슬라이스가 바뀔 때마다 해당 디스크가 교체되는 방식으로 전체 분산이 달성됩니다. 따라서 특정 디스크가 장애났을 때, 해당 슬라이스 내 모든 행의 재구성 쓰기가 동일한 `spare` 디스크 하나로 집중됩니다. 슬라이스를 넘어가면 다른 디스크가 그 역할을 맡습니다. 이 분산의 균등성은 `pool` 생성 시 `draidcfg`가 결정하는 `permutation map`의 품질에 달려 있습니다. `draidcfg`는 동일 구성에 대해 `permutation map`을 여러 번 랜덤 생성한 후, 특정 디스크가 `spare`를 과다하게 담당하는 `collision`이 가장 적은 `map`을 선택합니다.[^3] `9d:90c:1s` 기준으로 시뮬레이션(seed=42)한 결과는 다음과 같습니다.

| 항목      | 값         |
| :------ | :-------- |
| 전체 슬라이스 | 736개      |
| 이론적 평균  | 8.18회/디스크 |
| 최다 담당   | D11 — 13회 |
| 최소 담당   | D47 — 2회  |

  736을 90으로 나누면 8.18로 정수가 아니므로 완벽한 균등 분산은 수학적으로 불가능합니다. 따라서 어느 디스크에 장애가 나느냐에 따라 복구 시간에 어느 정도 편차가 발생할 수 있습니다.

## raidz2 vs dRAID2 — 장애 복구 동작 방식의 차이

  두 구현의 장애 복구 속도 차이는 물리 레이어와 파일시스템 레이어의 결합 방식에서 비롯됩니다. 특허 문서(US12547323)[^4]에 이 차이가 명확히 기술되어 있습니다.

> "Because of its dynamic structure, RAID-Z cannot rely on fixed geometric assumptions when restoring a failed disk. Instead, RAID-Z must traverse filesystem metadata to determine the structure of each block and its placement."

  `raidz`는 논리 블록 크기가 매번 다른 **dynamic stripe width**를 사용합니다. 이 때문에 장애 복구 시 어느 오프셋에 어떤 블록이 있는지 알기 위해 파일시스템 메타데이터 전체를 순회해야 합니다. 대형 어레이에서 이 순회가 랜덤 I/O로 이어져 장애 복구가 느려집니다.

  `dRAID`는 파일시스템 레이어와 물리 레이어를 완전히 분리하고 **고정 stripe width**를 사용합니다. 물리 레이어는 오프셋 0부터 끝까지 16MB 행 단위로 순서대로 처리하면 되므로, 메타데이터 순회 없이 **sequential rebuild**가 가능합니다. 이것이 장애 복구 속도 향상의 근본적인 이유입니다. 다만, 이 분리의 대가는 공간 효율입니다. 논리 블록이 물리 `stripe`에 딱 맞지 않을 경우 남는 공간을 0으로 패딩해야 합니다. 작은 블록이 많은 워크로드에서 공간 낭비가 심해지는 이유입니다.[^5] 또한 두 구성은 `vdev` 수 측면에서도 근본적인 차이가 있습니다.

| 구성     | write 파이프라인 | read 병렬도 |    장애 복구 방식     |
| :----- | :---------: | :------: | :-------------: |
| raidz2 |     9개      |  10개/그룹  |  healing (느림)   |
| dRAID2 |     1개      |   90개    | sequential (빠름) |

`raidz2 × 9`는 9개의 독립 write 파이프라인으로 write 성능이 유리하고, `dRAID2 × 1`은 90개 디스크 전체를 하나의 read 파이프라인으로 활용해 read 성능이 유리합니다.

| 풀 구성 B  | `d2-9d-1s` (dRAID2:9d:90c:1s)          |
| 실험 조건   | 200TB 채운 상태에서 디스크 1개 강제 장애 후 복구 시간 측정  |
### Rebuild vs Resilver

OpenZFS에서 복구 프로세스는 교체 대상에 따라 두 가지로 나뉩니다.

  - **rebuild** — `dRAID` 전용. `distributed spare`로 논리 재매핑. `spacemap`만 스캔하며 체크섬 검증 없이 순서대로 복구. `zpool status`에 `rebuilt`로 표시.
  - **resilver** — `raidz/dRAID` 공통. 물리 디스크 교체 후 블록 포인터 트리를 순회하며 체크섬 검증과 함께 복구. `zpool status`에 `resilvered`로 표시.

따라서 `dRAID`는 장애 발생 시 2단계로 복구가 진행됩니다.

  - **Phase 1**: 장애 직후 `distributed spare`로 논리 재매핑 → **rebuild**. 약 1초 만에 완료되어 풀이 즉시 `ONLINE` 상태로 복귀. `dRAID`에만 존재하는 단계.
  - **Phase 2**: 물리 디스크 교체 후 실질 데이터 복구 → **resilver**. `raidz2`와 `dRAID2` 모두 동일한 프로세스. 이것이 실질적인 복구 완료 시점.

`raidz2`의 경우는 `Phase 1`이 없고, 물리 교체 후 바로 `resilver`가 시작됩니다. 때문에 `resilver`가 완료되기 전까지 `pool`의 상태는 `DEGRADED` 상태로 표시됩니다.

`resilver` 진행 중 `fio random read/write`를 27.8시간 동안 병행한 결과입니다. `CIO`는 `Concurrent I/O` 의 줄임말로 편의상 `resilver` 중에 동시에 수행한 I/O임을 표현하기 위해 사용한 용어입니다

## 마치며

`dRAID`는 물리 레이어와 파일시스템 레이어를 의도적으로 분리함으로써 `sequential resilvering`를 가능하게 한 설계입니다. 그 대가로 고정 `stripe width`에 따른 공간 낭비와 write 성능 저하가 발생하지만, 대형 어레이에서의 장애 복구 시간 단축과 운용 편의성이 이를 보완합니다. 이번 포스트에서는 `dRAID`에 대한 기본 구조를 파악하고 이를 바탕으로 간단하게 `raidz2`와 `dRAID2` 비교 실험을 진행하였습니다.

특히 이번 실험은 `9d:90c:1s` 단일 구성에 대한 결과입니다. d값(data disks per group)을 변경하면 슬라이스 크기, IOPS, usable capacity, rebuild 속도가 모두 달라집니다. 추후 `4d:90c:4s`, `6d:90c:4s` 등 다양한 구성에 대한 추가 실험을 통해 최적의 구성도 탐색해볼 예정입니다.
